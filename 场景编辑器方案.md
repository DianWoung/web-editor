# A1 开发蓝图

**项目名称：机房数字孪生 Web 场景编排器 + 展示前端**

## 0. 先给结论

本项目建议采用 **A1 路线：React + React Three Fiber + Three.js**，做成一套 **Web 化数字孪生前端**。这条路线和现有方案完全一致：前端应用层用 React，三维渲染层用 React Three Fiber + Three.js，目标是便于浏览器部署、快速迭代和跨项目复制。

建模方法不走“高精 BIM 前置”，而是以 **CAD 二维图纸作为布局输入**，采用 **设备库 + 管线规则化建模** 的方式构建轻量三维机房场景，并直接建设到 **L2 设备语义层**。

------

# 1. 我们做这个选择的背景信息

## 1.1 为什么不直接走“纯 AI 自动建模”

当前实际问题已经很明确：
**不是模型本身不能看，而是设备摆位、朝向、管线连接关系不可靠。**

这说明问题不在于“渲染引擎不够强”，而在于缺少一套可约束的生产方法。现有方案已经把这个边界定义清楚了：

- 首期目标是 **运营级数字孪生**
- 重点先满足 **空间表达、设备交互和数据绑定**
- 不以高精度 BIM 为前提
- 所有三维对象都应具备设备语义、系统归属、测点挂接与策略关联能力
- 从第一期开始就沉淀资产命名、设备编码、场景组织、管线规则和接口规范，形成可跨项目复用的建设标准

所以这里不能把“AI 直接猜出正确机房”当成主路线。

## 1.2 为什么不优先走 Unity / 重引擎路线

Unity 能做强展示，但本项目当前主诉求是：

- 电脑浏览器为主
- 后续要可复制交付
- 除了 3D 场景，还要承载图表、详情、告警、模式解释、AI 建议
- 首期要快速形成单机房 MVP

而现有前端方案已经明确：
**采用 Web 化数字孪生前端**，前端承担展示、解释、分析与受控交互；后端控制系统继续承担采集、联锁、执行与保护。

这意味着核心竞争力不是“做一个更炫的 3D 程序”，而是把 **机房空间、设备对象、实时数据、告警、控制模式、节能结果和 AI 建议** 统一组织成一个可视化入口。

## 1.3 为什么选 A1

A1 的本质不是“做一个 3D 页面”，而是做一个：

**Web 场景编排器 + 展示前端一体化方案**

它符合当前项目的四个现实约束：

1. **浏览器优先**
   方案明确要求浏览器部署便捷、可快速迭代、可跨项目复制。
2. **先运营、后高精**
   先把空间表达、主要管线、系统逻辑、设备对象化绑定做对，再逐步补高精资产。
3. **以 CAD 为布局输入，不依赖完整 BIM**
   方案明确采用 CAD 二维图纸作为布局输入。
4. **必须直接做到 L2 设备语义层**
   不是简单的几何展示，而是建立设备对象、系统对象、数据对象之间的稳定映射。

# 2. A1 的目标定义

## 2.1 项目目标

建设一套面向机房 AI 节能控制场景的前端产品层，首期完成：

- 单机房三维总览
- 核心设备与主要管线展示
- 设备详情交互
- 设备、测点、系统、告警、策略的对象化绑定
- PC 操作版
- 大屏展示版

这和现有首期交付范围完全一致：首期建议以单机房 MVP 为切入范围，优先验证三维展示效果、数据挂接稳定性与工程使用价值。交付项包括三维总览页、设备详情页、语义映射配置、设备库首版和实施规范。

## 2.2 系统边界

前端只负责：

- 三维场景
- 设备交互
- 状态呈现
- 模式解释
- 图表与看板
- AI 建议展示
- 受控命令入口

底层执行、安全联锁、设备保护仍由 PLC / DDC / 原厂控制器承担。

这也必须和总方案保持一致：
**AI 做上层优化，PLC 做底层执行，原厂控制器做设备保护。**

# 3. A1 的总体架构

## 3.1 技术栈

- 前端框架：React + TypeScript
- 构建工具：Vite
- 三维引擎：Three.js
- React 三维封装：React Three Fiber
- 状态管理：Zustand
- 图表：ECharts 或等价方案
- 3D 资产格式：GLB
- 数据接口：统一 API / Mock API

## 3.2 四层架构

### 第一层：设备资产层

职责：

- 存放标准设备模板
- 管理模型、端口、缩略图、资产元数据
- 沉淀命名、导出、挂点规范

每个资产建议包含：

- `model.glb`
- `asset.json`
- `ports.json`
- `thumbnail.png`

### 第二层：场景编排层

职责：

- 设备拖拽摆位
- 设备旋转与朝向调整
- 端口显示
- 端口到端口连接
- 管线规则化生成
- 场景保存/加载

这是 A1 的核心，解决当前“摆放和管线完全不行”的问题。

### 第三层：语义映射层

职责：

- 绑定设备编码
- 绑定测点
- 绑定系统归属
- 绑定上下游关系
- 绑定告警
- 绑定策略与模式解释

这部分直接对应 L2 语义层要求。

### 第四层：展示输出层

职责：

- 输出 PC 操作版
- 输出大屏展示版
- 驱动总览页、详情页、看板、镜头巡航、AI 建议展示

# 12. 实施要点与风险（供排期与验收对齐）

本章把前文未展开但会影响落地的事项，收成可执行的清单，便于评审工时、接口和验收标准。

## 12.1 CAD → 场景（必须定一条最小流程）

首期建议明确 **一种** 可复制的路径，避免“有编辑器、无输入”：

- **坐标与单位**：世界坐标原点、轴向（如 Y 向上）、长度单位（建议米），与 CAD 导出/描图约定一致。
- **房间边界**：多边形或外包矩形如何进入 `scene`（手工描边 / DWG-DXF 辅助 / 仅示意墙线），谁负责校准。
- **设备初值**：设备清单（类型、编码、大致位置）是否允许来自表格/JSON，再在 3D 中精调，以控制实施人天。
- **变更机制**：现场改机、改管后，场景如何回写与版本留痕（见 12.3）。

## 12.2 数据与后端契约（L2 与实时展示）

- **标识主从**：`deviceId`、现场设备编码、测点 ID、告警 ID 的命名空间与对照表归属（建议后端或配置服务为权威源）。
- **实时通道**：WebSocket / SSE / 轮询择一写清；推送频率上限、断线重连、时间戳与质量位（若有）如何展示。
- **Mock → 真数据**：统一 API 形态与错误码；切换环境时绑定配置不硬编码在前端 bundle。
- **策略与模式**：`strategyVersion` 与后端策略包版本对齐；升级时的兼容或强制重绑规则需约定。

## 12.3 场景与资产的持久化、版本

- **存储位置**：场景 JSON 存对象存储 / 业务库 / 本地文件导出中的哪一种为“发布源”，开发/预发/生产如何隔离。
- **Schema 版本**：`scene` 与 `asset.json` 带版本号；旧版场景的迁移策略（脚本或兼容读取层）。
- **协作**：编排页是否单人锁编辑；若多人，冲突解决策略（最后写入 / 分支合并）首期可明确“不做”以免范围膨胀。

## 12.4 安全与权限（含受控命令入口）

- **角色**：实施人员、运维、参观大屏等角色的只读/可编/可下发划分。
- **命令路径**：浏览器 → 后端鉴权与二次校验 → 执行层；审计日志、限流、超时与拒动回显。
- **大屏与公开展示**：默认只读、隐藏敏感测点或聚合展示的策略（按项目需要勾选）。

## 12.5 性能与验收指标（建议写进首期 DoD）

按单机房目标规模（设备数、管线段数、贴图体量）约定 **可测** 条目，例如：

- 目标客户端（如常见办公本集显）下交互帧率或操作响应上限；
- 首屏与场景加载时间上限（可含懒加载与分级加载策略）；
- 大屏基准分辨率与安全区、3D 画布 `devicePixelRatio` 策略。

技术侧预留：**实例化/合并几何、LOD、按需加载**，避免后期整体重构。

## 12.6 编辑器工程化（与“摆位可信”配套）

- **撤销/重做**、复制实例、对齐/吸附等按优先级分期，但至少保证“误操作可恢复”或明确首期不做并承担培训成本。
- **校验**：发布前绑定完整性检查（未映射测点、悬空管线、重复 ID 等）可先做规则列表，再逐步实现自动拦截。
- **资产升级**：`assetId` 对应 GLB/端口变更时，旧场景的兼容或一键重绑流程。

## 12.7 质量保障

- 关键路径 E2E（加载场景、保存、总览点击进详情、大屏巡航）；
- GLB 资源规范（面数上限、压缩、命名）与验收清单；
- 可选：核心页面的可视化回归（按需）。

## 12.8 已识别重大风险与缓解摘要

| 风险 | 缓解方向 |
|------|----------|
| CAD 数字化与人工编排成本高 | 固定最小流程 + 表格/JSON 初值 + 工时估算纳入里程碑 |
| 正交管线路由与“不穿包围盒”难度被低估 | MVP 允许简化规则或“检测高亮 + 手工改路径”；自动避障分期 |
| WebGL 性能与大屏分辨率压力 | 早做假数据压测 + Instancing/LOD/合并 + 大屏基准分辨率约定 |
| 现场点表/策略变更导致绑定失效 | 配置版本化 + 发布前校验 + 环境分册 |
| 控制入口引发安全与验收争议 | 权限、审计、后端校验写进范围与接口设计，避免联调末期补洞 |

**说明**：无障碍、国际化等若首期不纳入，建议在评审会上 **显式勾选“不做”**，避免隐含预期。

# 13. 附录：技术生态选型与对标参考

本章补充与 **A1（R3F + Three.js）** 配套的开源生态、可选增强与行业对标，供架构评审与分期立项使用；**不改变**前文已定的首期边界（运营级、非高精 BIM、L2 语义优先）。

## 13.1 React Three Fiber 生态（Poimandres）

| 包 | 作用 | 与本文档的对应 |
|----|------|----------------|
| `@react-three/fiber` | React 与 Three.js 的声明式桥接 | 与第 3 章技术栈一致，作为 3D 编排与展示的核心底座 |
| `@react-three/drei` | 常用 3D 原语与控制器 | **OrbitControls**、**Grid**、**TransformControls**、**useGLTF**、**Environment**、**Html**（屏幕空间标签）、**Outlines / Select**（选中高亮）等，可覆盖阶段 1 大部分交互与展示需求 |
| `@react-three/xr` | WebXR（如 VR 巡检） | **远期可选**：若大屏或展厅需要 VR 漫游，可在同一 R3F 树上渐进接入；**不要求**纳入首期 MVP |

**建议**：在 drei 已满足轨道控制、模型加载、辅助网格的前提下，按需增量引入 **Html**（设备旁标签/告警角标）、**Outlines**（统一选中样式）、**Environment**（环境光与反射一致性）；避免为「观感」一次性堆叠过多依赖。

## 13.2 属性面板：Leva / Dat.GUI

- **定位**：快速生成可调参数面板，适合开发期调试相机、灯光、材质、管线参数等。
- **与第 4.1 节「右侧属性面板」的关系**：Leva/Dat.GUI 更适合 **工程调试**；面向实施/运维的 **正式属性面板** 仍建议保留 **表单 + 校验 + 与后端字段一致** 的产品形态，避免调试 UI 直接暴露给交付场景。
- **落地方式（建议）**：例如通过 `?debug=1` 或内部账号挂载 Leva，与生产编排界面分离。

## 13.3 管线生成与 2D 拓扑：React Flow

- **背景**：三维空间正交自动布线 **没有**「开箱即用、通用完美」的单一 JS 库；MVP 可先规则化生成 + 冲突高亮 + 手工修正（见第 6 章、第 12 章）。
- **React Flow 的价值**：在 **二维** 上管理「节点—连线」拓扑（多系统、支路、阀门逻辑等），可读性与编辑效率常优于纯 3D 点选。
- **与 3D 的映射**：将 React Flow 导出的 **连接 JSON**（`from`/`to` 或节点边列表）映射为本文第 5.1 节的 **`pipe.from` / `pipe.to`（`deviceId.portId`）**，再由前端或工具生成 3D 正交路径。
- **必须约定**：**单一真相源**——以 Flow 为准或以 3D 连线为准需写清；避免双写导致拓扑与几何长期不一致。
- **分期**：阶段 1 可维持「双端口点选建连」；连接关系复杂化后再引入 React Flow 成本更可控。

## 13.4 BIM / CAD 轻量化：That Open（原 IFC.js）

- **与「不做高精 BIM」的关系**：不将施工级 BIM 作为目标时，仍可将 **轻量 IFC** 作为 **可选输入**，用于抽取 **机房边界、墙体轴线、构件大致位置与属性**，再落入 `scene` 的房间边界或参考层。
- **That Open Company（IFC.js 路线）**：Web 端解析 IFC 的常用开源路径之一，适合「只取边界与少量构件」的裁剪用法。
- **工程建议**：将 IFC 解析与主编排应用 **分包或独立微工具**，控制 bundle 体积与 WASM 依赖对日常开发的影响。

## 13.5 开源/平台对标（L2 与数据模型）

| 对标对象 | 形态 | 可借鉴点 |
|----------|------|----------|
| **FUXA** | Web 端开源 SCADA（偏 2D） | 数据绑定、Mock 协议、告警与工程 JSON 组织方式，可作为 **L2 设备语义层** 落地时的**结构参考**（非替代 3D 编排器） |
| **ThingsBoard** | 开源 IoT 平台 | **Device Profile**、**Telemetry**、设备属性与遥测模型，有助于完善第 5.1 节 **`semanticBinding`** 与后端点表、告警 ID 的命名及分层 |
